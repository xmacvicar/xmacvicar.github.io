# Editorial & Technical Writing Standards

These guidelines apply to all documentation in this portfolio. Follow them when reviewing or editing any article.

## Voice & Tone

- Use active voice throughout. Rewrite passive constructions as direct instructions.
- Use imperative verbs for instructions ("Tap", "Select", "Enter"): not "You can tap" or "You should select."
- Avoid hedging language: "should," "may need to," "you might want to": be direct.
- Remove filler phrases: "Please note," "It is important to," "In order to."
- "whether" not "whether or not."
- No exclamation marks in technical content.

## Structure

- One H1 per page (the page title). All other headings are H2 or below.
- Use numbered lists for sequential steps.
- Use bullet lists for non-sequential items (settings, options, reference lists).
- Tip/Note/Warning callouts contain a single focused point: not multi-step instructions.

## UI Elements

- Bold all UI element names: buttons, menu items, tabs, toggles (e.g. **Sign In**, **Setup**, **Check Out Mode**).
- Use "tap" not "click" for mobile app instructions.
- Use "select" for dropdowns and list items.
- Refer to UI elements exactly as they appear in the interface: match capitalisation.

## Formatting

- Capitalise all terms consistently within a list (e.g. definition lists: all labels title case).
- Do not use em dashes (—). They are a common AI writing tell. Use a colon or rewrite the sentence instead.
- For definitions in bullet lists, use a colon: **Label**: description.
- Measurements and ranges: use a hyphen with spaces (6-8 inches).

## Specific to Event Scan App docs

- The product is called "Event Scan": not "the app" on first reference per page.
- Sync-related instructions: users do not need to sync manually; the app syncs automatically on a reliable connection.
